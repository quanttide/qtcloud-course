# 量潮课程云架构设计

## 领域定位

量潮课程云是课程领域的落地系统，分为两个子领域：**课程单位**（课程研发）和**教学单位**（教学实施）。它与量潮课堂（组织领域）通过交付模式关联，但课程域本身不感知组织侧信息。

```
┌──────────────────────────────────────┐
│           课程单位子领域               │
│  (课程研发：教什么、用什么教)           │
│                                       │
│  Program(专业) → Course(课程) → Lesson(课时) │
└──────────────────┬───────────────────┘
                   │ 引用/实施
                   ▼
┌──────────────────────────────────────┐
│           教学单位子领域               │
│  (教学实施：谁来教、怎么教)             │
│                                       │
│  Class(教学单位)                      │
└──────────────────────────────────────┘
```

## 课程单位子领域

关注课程研发，定义教学内容的结构：

| 实体 | 职责 | 示例 |
|:----|:----|:----|
| Program | 专业，顶层教学计划 | 大数据微专业 |
| Course | 课程，教学单元 | 数据工程 |
| Lesson | 课时，教学内容的最小组织单元 | 课时1 |

## 教学单位子领域

关注教学实施，组织学员和教学资源：

| 实体 | 职责 | 示例 |
|:----|:----|:----|
| Class | 教学单位，学员共同学习的组织 | 浙理班级 |

## 关键关系

- **Program 包含多个 Course**：项目集定义教学计划的范围。
- **Course 包含多个 Lesson**：项目由多个课程研发单位组成。
- **Class 引用 Program 或 Course 的内容进行教学实施**：教学单位不重新定义教学内容，直接使用课程单位的产出。
- **Class 本身不感知组织**：同一个 Class 可被不同 Contract 覆盖。

Class 与组织侧的关系由量潮课堂通过 Contract 和 Enrollment 管理，课程云不关心谁买单、谁学习。

## 设计规则

1. **课程单位子领域定义内容，教学单位子领域负责实施**——Class 不自主开发课程，直接使用 Program / Course / Lesson。
2. **课程域不引用组织域**——Class 不知道谁买单，Course 不知道谁在学习。
3. **数据驱动加惰性演进**——优先落地 Program 和 Class，Course 和 Lesson 延后细化。

## 不解决的问题

- **教学质量管理**：评分、反馈、作业批改——属于教学评估领域。
- **排课与资源调度**：教室、设备、时间冲突检测——独立排课模块。
- **AI 辅助教学**：内容生成、智能批改——作为独立能力层引入。
